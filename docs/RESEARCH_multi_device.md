# Research: multi-device and multi-person DeckBoss

Status: **Phase 2+ candidate, not scheduled.** This document pressure-tests what it would actually take for DeckBoss to move from "one phone, one captain" to "more than one device or person logging the same fishing operation." It is a design and research artifact, not a plan to implement now. Code references are to the state of the codebase when this was written.

## Scope boundary

This stays inside the logbook. The sibling `cocapn-foundation` repo owns the bigger autonomy/steering/voice-command vision; this document does not wander into autopilot, NMEA integration, Bluetooth mesh "hot bite" sharing, or anything else that changes how the boat is driven. The question is narrower: *the same catch/event gets recorded by more than one phone or more than one person — what breaks, what needs to change, and is it the right next phase?*

## 1. Identity — who recorded this, and does it matter?

### What exists today

`LogEntry` (`src/core/types/log-entry.ts`) has no field for device, person, or author of the *entry* itself. The only identity-like concept is the optional `author` on `Correction`, added by `docs/DESIGN_transcript_as_event.md`, which distinguishes:

- `{ kind: "human" }`
- `{ kind: "model", engine: "webspeech" | "whisper-1" }`

That tells you *what kind of agent* produced a correction, but not *which* human or *which* phone. `buildEntry()` (`src/core/tensor-log/entry-builder.ts:58`) tags the first transcript correction as model-authored; `buildAmendCorrection()` defaults new corrections to human-authored. There is no device fingerprint anywhere.

### Two devices of one person

For this case, "who" genuinely does not matter for correctness. The merge primitive (`src/core/sync/conflict-resolver.ts:17`) unions corrections by id, dedupes, and sorts by `created_at` with an `id` tiebreak. The two phones are interchangeable loggers; the only thing that has to be unique is the entry id (UUIDv4, generated by `newId()` in `src/utils/id.ts`), and the only thing that has to be monotonic-ish is correction ordering.

What *would* matter, even for one person with two phones, is telling the user which device still has an audio blob that the other one lacks. That is a sync/availability question, not an identity question, and it falls under §3 below.

### Two or more people

The moment there are multiple *people*, identity becomes load-bearing for at least three reasons:

1. **Attribution in the UI.** Two deckhands correcting the same entry is fine structurally, but a captain reviewing the log later reasonably wants to know who said "actually it was coho, not chinook."
2. **Permission boundaries.** A boat owner seeing a crew's aggregate logs is a different visibility model than a shared crew logbook. Today's BYOK storage model means "everyone points at the same cloud folder" — which is either full visibility for everyone who has the credentials, or no visibility, with no granularity in between.
3. **Conflict suspicion.** If two humans amend the same field to different values, the current fold just picks one by timestamp+id. At crew scale that becomes a silent override of one person's input, not a harmless merge of two devices.

### A minimal identity model

The product has zero account system by design, and that is a feature, not a bug. A minimal, BYOK-compatible identity layer could be:

```ts
// Add to CorrectionAuthorSchema / a new entry-level field
deviceId: z.string().uuid().optional(),   // locally generated, persisted in AppConfig, never rotated
personLabel: z.string().max(64).optional(), // free text: "Matt", "Relief skipper"
```

- **`deviceId`** is generated once per browser profile and stored in IndexedDB (in `AppConfig`, which is already local-only). It is not a login; it is just a stable label so two phones don't look identical in a merged log. Because it lives in `AppConfig`, it never crosses a `StorageAdapter` boundary, so BYOK is preserved.
- **`personLabel`** is also local-only config, optionally set in Settings. It travels with corrections/entries in the synced Markdown frontmatter so attribution survives across devices.
- **No central auth, no key exchange, no invitations.** If a crew wants a shared log, they share the storage credentials/point everyone at the same folder, exactly as they would share a paper logbook or a shared Drive folder.

This is enough for attribution and basic filtering ("show me entries recorded by Matt"). It is *not* enough for cryptographic proof of who did what, and it is not enough to prevent someone from lying about their `personLabel`. That matches the project's existing regulatory stance: DeckBoss is a personal log, not evidence-grade.

### Honest caveat: identity is cheap, but it makes other problems visible

Adding `deviceId`/`personLabel` is a small schema change. The reason to hesitate is that it surfaces problems the current model can currently ignore: if two people retract the same entry for different reasons, the fold just marks it retracted and keeps the most recent reason. With identities attached, a user can now *ask* "who retracted this and why?" — and the data model does not yet have a good answer for "two people disagreed." The schema supports the question before the product has decided how to answer it.

## 2. Conflict resolution at genuine multi-person scale

### The current mechanism

`mergeEntries()` in `src/core/sync/conflict-resolver.ts` does three things:

1. Concatenate local and remote `corrections`.
2. Deduplicate by `correction.id`.
3. Sort by `created_at`, then by `id` lexicographically.

`applyCorrections()` (`src/core/tensor-log/entry-builder.ts:113`) then folds in that order, with later amendments winning field-by-field. Retraction is sticky once it appears. `invariants.ts` enforces that capture facts (`id`, `timestamp`, `gps`, `audio`, `source`, `thread_id`, `version`, `transcript`, `entities`, `tags`) never change and that corrections are append-only and immutable.

### Why this is safe for two devices

The stress tests in `tests/unit/conflict-resolver.test.ts` and `tests/unit/concurrent-edit-scenarios.test.ts` establish two good properties:

- **Idempotence:** merging the same correction set twice produces the same result.
- **Argument-order independence:** `mergeEntries(A, B) === mergeEntries(B, A)` for the same inputs.

Those hold because deduplication and deterministic sorting erase whatever order the corrections arrived in.

### Why it is not a CRDT at scale

A CRDT guarantees that any two replicas that have seen the same set of operations converge to the same state, regardless of order. The current model gets close but fails on a subtle point: **corrections are not semantically commutative for the fields they replace.**

`transcript`, `entities`, and `tags` are overwritten wholesale by the latest amendment in sort order. If two devices create corrections at nearly the same wall-clock time, the sort falls back to lexicographic `id` comparison. That tiebreak is deterministic and stable — but it is also *arbitrary*. The result depends on which random UUID happened to sort first, not on which correction is "better."

At two-device scale this is a corner case: two people have to edit the same entry within the same clock millisecond (or with clock skew that collapses their timestamps). At five-device crew scale it becomes routine: multiple people tag, correct species, or add notes to a popular entry. The system will silently pick a winner by UUID lottery.

Two other CRDT gaps:

- **Clock skew.** `created_at` is set by the originating device's clock. If Device A's clock is two minutes fast, its corrections will consistently sort ahead of Device B's, even when B's were created first. The system remains deterministic, but deterministically wrong from a human point of view.
- **No causal metadata.** There is no vector clock, no logical timestamp, and no parent pointer. The model cannot distinguish "A happened before B" from "A and B happened concurrently." For a field-replacement fold, that distinction is exactly what would let the UI say "these two corrections conflicted — pick one" instead of silently ordering them.

### What would be needed to make it rigorous

If the product ever needs a genuine mathematical convergence guarantee for a crew, the corrections model would need to become one of these:

1. **CRDT for each editable field.** Tags could become a set-union CRDT (add/remove pairs with tombstones). Transcript could stay last-writer-wins but with a Lamport timestamp or hybrid logical clock instead of wall-clock+random-id.
2. **Operational transform lite.** Keep corrections as operations, but add a per-entry vector clock so the fold can detect true concurrency and surface it in the UI.
3. **Accept arbitrary ordering for non-concurrent edits, flag concurrent edits for review.** This is the pragmatic middle path: use wall-clock time as a best-effort ordering, but when two corrections have timestamps within some small delta and touch the same field, mark the entry as having a "review" state instead of silently picking one.

My honest assessment: the current model is **deterministic and safe (no data loss), but not a true CRDT and not fair at crew scale.** For two-device-same-person it is good enough. For multi-person it is a foundation that needs either causal metadata or a UI for resolving simultaneous field edits.

## 3. Visibility and ownership — does shared BYOK storage already work?

### The happy path: point everyone at the same bucket/folder

DeckBoss's storage model is deliberately backend-agnostic. Every adapter implements the same `StorageAdapter` interface; every entry is a Markdown file at `DeckBoss/YYYY/MM/DD/{id}.md`; audio lives at `.deckboss/attachments/{id}_audio.{ext}`; the manifest is `.deckboss/manifest.json` (`src/core/storage/interface.ts:82-96`).

If every device is configured with the same backend and the same location, the read side basically works today:

- **R2 / Oracle OCI:** Everyone with the same bucket credentials and endpoint sees the same objects. `pullRemoteEntries()` (`src/core/sync/sync-engine.ts:142`) reads the manifest, iterates `.md` files, parses them, and merges into local IndexedDB.
- **Google Drive:** More subtle. The adapter uses `drive.file` scope, which restricts the app to files it created *or* files in folders it can access. Multiple DeckBoss instances using the same Google account and the same `folderId` should see the same files, because the files belong to that account. Multiple *different* Google accounts sharing a Drive folder is a different story: `drive.file` would not let the app see files created by another account unless those files were explicitly opened/shared with the app's OAuth client. This is a real limitation: "shared Google Drive folder" is not as frictionless for DeckBoss as it is for humans browsing Drive.
- **Local ZIP export:** Not a live shared backend; it is a snapshot/import mechanism. Two crew members passing a ZIP back and forth would create merge conflicts the current code does not handle.

### What breaks under concurrent sync

The most important finding: **the manifest is a single write-last-wins file that each device regenerates from its own local state.**

`syncNow()` (`src/core/sync/sync-engine.ts:288`) does:

1. `pushAllLocalEntries()` — enqueue every local entry for upload.
2. `pullRemoteEntries()` — read the remote manifest, download every entry listed, merge into local IndexedDB.
3. `reconcileAudio()` — requeue unverified local audio.
4. `refreshManifest()` — write a new manifest containing *only* what this device currently has locally.

Step 4 is the problem. If two devices sync at the same time:

- Device A reads manifest M₀.
- Device B reads manifest M₀.
- Both upload their local entries.
- Both pull and merge remote entries into local IndexedDB.
- Device A writes manifest Mₐ = everything in A's local DB.
- Device B writes manifest Mᵦ = everything in B's local DB.

If A and B had exactly the same entries after the pull, Mₐ and Mᵦ are identical and the race is harmless. But if one device failed to download an entry (network hiccup, parse error, skipped corrupt file), or if a third device C uploaded something after A's pull but before A's manifest write, the last manifest write can *lose* entries that are still present in the cloud. The files still exist, but `pullRemoteEntries()` won't discover them without a manifest listing.

This is not a data-loss bug (the `.md` files are still there), but it is a **discovery bug**: a device can stop seeing entries that actually exist in shared storage. At one-device scale this never matters. At two-device scale it is rare. At crew scale it becomes likely enough to be a real support burden.

Other shared-storage rough edges:

- **Manifest write is not atomic with entry uploads.** A device can upload ten entries, crash before refreshing the manifest, and leave those entries invisible to other devices until the next successful sync. `pushAllLocalEntries()` enqueues jobs and then drains them, but `refreshManifest()` only runs after `drainQueue()` returns. A crash or tab close between the two leaves orphan files.
- **No per-device "I have verified my audio" state travels.** `markAudioVerified()` writes to a device-local IndexedDB store (`src/core/storage/local-db.ts:164`). That is the correct design — verification is a fact about a device, not the capture — but it means every device re-verifies audio for itself. That is fine; the alternative (trusting another device's verification) would be dangerous. But it means the first sync after a new device joins may re-upload audio that another device already archived.
- **Audio path collisions.** `audioPath(entryId, ext)` (`src/core/storage/interface.ts:94`) is deterministic by entry id. If two devices independently record *different* events that happen to get the same id (UUID collision), their audio blobs would collide. The probability is negligible and not worth designing around.
- **No concept of "owner" of an entry.** In a shared folder, anyone with credentials can retract or amend anyone else's entry. That is consistent with a paper logbook on the bridge, but it is not consistent with "boat owner sees, crew does not edit."

### The registry.ts / adapter assumptions

`registry.ts` (`src/core/storage/registry.ts`) caches the adapter instance by config. There is no assumption that only one device exists, but there is an assumption that **one browser profile = one adapter instance = one user's credentials**. The cache key is built from the storage section of `AppConfig` only; transcription settings do not invalidate it. That is correct for multi-device: each device has its own cache and its own credentials.

The adapter code itself is stateless with respect to entries; it does not assume single-device usage at the object level. The single-device assumption lives higher up, in `sync-engine.ts`'s manifest strategy.

## 4. What this explicitly is not

This document does not propose:

- A DeckBoss-operated backend or fleet learning layer (already rejected in `ROADMAP.md`).
- Real-time collaboration, cursors, or live presence.
- Cryptographic signatures, tamper-evident logs, or chain-of-custody.
- Voice-command steering, autopilot, NMEA, sensors, or any `cocapn-foundation` scope.

The scope is "more than one device/person logging the same fishing operation," and even within that, this document recommends a deliberately narrow first slice.

## 5. Recommendation — split it, and probably don't make it the next phase

### Three distinct slices

1. **Narrowest: accidental multi-device becomes intentional.**
   - Add a stable `deviceId` to `AppConfig`, generated at first run.
   - Attach it to new corrections (and optionally to new entries) as metadata only — no change to merge semantics.
   - Fix the manifest race so that shared BYOK storage is actually reliable when two devices sync.
   - This is the smallest slice worth banking. It makes the existing two-device safety net explicit and testable.

2. **Medium: same-boat crew logbook.**
   - Add `personLabel` to corrections/entries.
   - Add a UI affordance for "entries by this person/device."
   - Document the shared-storage setup (Drive folder, R2 bucket, OCI bucket) and its limitations.
   - Accept that merge semantics stay timestamp+id for now, with identity only for attribution.

3. **Largest: true multi-person conflict fairness.**
   - Add causal metadata (vector clock or Lamport clock) to corrections, or build a per-field CRDT.
   - Surface concurrent edits in the UI for human review instead of silently picking by UUID.
   - This is a quarter-scale architecture change, not a week-scale feature.

### What to bank first

The narrowest slice is worth doing, but **not on its own as a Phase 2 kickoff.** The reason is that the manifest race is a live bug the moment the registry.ts adapter-caching fix made real sync actually work. It is already possible for two devices to point at the same R2 bucket today and have one device's manifest write hide the other's entries. That should be fixed regardless of whether multi-device is a named feature, because the product's own backup story ("your cloud storage is the canonical archive") assumes the archive is discoverable.

The concrete minimal work:

- Make `refreshManifest()` merge with the existing remote manifest instead of replacing it. The manifest should be the union of all files that exist in the storage root, not just the files this device currently remembers.
- Add `deviceId` to `AppConfig` and stamp it on corrections. This is a one-field schema addition with no behavior change other than attribution.
- Add one or two integration tests that simulate two devices syncing into the same `LocalZipAdapter` and verify no entries are lost from the manifest.

### Whether this should be the next real phase

**Honest answer: probably not.** The `ROADMAP.md` "actual top priority" is correct: before designing multi-person logbooks, the project needs real fishermen using the app. Multi-device support is a real capability, but it is currently an *accident* of the corrections design, not a validated user need. The highest-leverage next phase is still field validation of the core recording/transcription loop.

That said, the *accident* should be shored up. The manifest race is a latent bug in the current single-device-plus-backup story; fixing it is defensive maintenance, not feature work. Identity metadata is cheap and unlocks clearer attribution when the field evidence finally arrives. Both are worth doing opportunistically — for example, the next time `sync-engine.ts` or `AppConfig` is touched for another reason — rather than scheduling a dedicated "multi-device phase" before the product has users who need it.

### Gating conditions for the larger work

Do not schedule the medium or large slices until:

1. At least one field-testing crew has actually asked for shared logs, or has clearly hit the "I recorded this on my phone and my deckhand recorded the same thing on his" case enough to matter.
2. The manifest race is fixed and has been tested under simulated concurrent sync.
3. There is an explicit decision on audio retention/eviction (`ROADMAP.md` still defers this), because multi-device audio availability is the first place users will feel a broken promise.

Until then, treat multi-device/multi-person as a well-understood future expansion, not the next milestone.
